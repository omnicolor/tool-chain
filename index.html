Here's a typical development layout.
  • Coders write their tests and code
  • They submit it to their source control
  • Continuous integration sees the changes and does its thing
  • If it fails, it kicks it back to the coders for more testing and coding.
  • If it passes, the new code is uploaded automagically to staging.
  • QA runs on staging and if it blesses the build, pushes it to production.
The first step is writing tests and code. Other sessions spend their entire time talking about these topics, so I'm not going to go into that here except to say that most of this toolchain requires good tests to be effective.
 
The second step is submitting to source control. You're going to hear a lot of people tell you that you have to use git. For the purposes of this discussion, you can use whatever source control system you want. If you're setting things up for the first time, you should probably use git. But using source control in the first place is far more important than which system you choose.

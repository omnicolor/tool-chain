Here's a typical development layout.
  • Coders write their tests and code
  • They submit it to their source control
  • Continuous integration sees the changes and does its thing
  • If it fails, it kicks it back to the coders for more testing and coding.
  • If it passes, the new code is uploaded automagically to staging.
  • QA tests staging and if it blesses the build, pushes it to production.

The second step is submitting to source control. You're going to hear a lot of people tell you that you have to use git. For the purposes of this discussion, you can use whatever source control system you want. If you're setting things up for the first time, you should probably use git. But using source control in the first place is far more important than which system you choose.
 

So what goes into source control? Ev-er-y-thing.

Disk space is cheap, so →

Don't throw anything away. All code and configuration should be saved.

I use two repositories: one for code, one for config.

Code is the same in all of your environments, except you don't push tests to staging or production.

Configuration changes between environments. Your deploy script pulls from the config repository depending on where you're pushing to.

If you trust your developers, anyone can write to any repository.

If you don't trust your developers, nothing in this talk will save you.

Next up is your continuous integration server.

This is where most of the toolchain magic shows up. Your CI server should be really strict and really demanding of code. It should run all of your tests and fail the build if anything is wrong.

The philosophy is that it's better to fail here quickly than waste QA's time checking a broken build on staging.

but if the build does break, you've got to drop everything and fix it so it doesn't slow down your team.
